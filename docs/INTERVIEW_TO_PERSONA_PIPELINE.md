## 1. Core Value Proposition

**The value of personas is SCALE, not fidelity to any single interview.**

A UX researcher with 20 interviews can't realistically simulate how those 20 people would react to 5 different feature concepts. The insight is locked in transcripts. Kynd's job is to:

1. Extract what's OBSERVABLE in the interviews (pain points, goals, values, feature desires)
2. Model the distribution of those signals across the interview pool
3. Sample N coherent synthetic personas from that distribution (where N > interview count)
4. Let the user chat with individual personas, tweak their psychographics, and run cohorts against features

**20 interviews → 40+ personas. Each persona is a coherent individual, not a stereotype.**

## 2. Why Not Extract Personality Traits

The academic literature (Zhu et al., 2025) establishes that zero-shot LLM extraction of Big Five personality traits from raw interview transcripts achieves Pearson correlations of r < 0.26 against ground truth. Even fine-tuned models plateau at r ≈ 0.35-0.40.

**Why this happens (first principles):**

- A 30-minute interview is a conversational performance, not a personality assay
- The interviewee manages impression, responds to interviewer rapport, and operates in a transient emotional state
- The LLM cannot differentiate between transient affective states (stress about the topic) and stable personality traits (high Neuroticism)
- The LLM has no model of the interviewer to subtract their influence

**The solution is NOT to try harder at extraction. It's to change what we extract.**

### What We Extract Instead

| Signal | Reliability | Source |
|--------|------------|--------|
| Pain points | High | Direct quotes: "I hate when X happens" |
| Goals / Jobs-to-be-Done | High | "I'm trying to accomplish Y" |
| Context (role, industry, team size) | High | "I work at a Z company in industry W" |
| Expressed values | High | "It's really important that..." |
| Feature desires | High | "I wish it could do X" |
| Decision patterns | Medium-High | "I always check with my boss first" |
| Communication style | High | Formal/casual, technical/business language |

All of these are DIRECTLY OBSERVABLE in the interview text. You can point to the exact sentence.

### What Gets Assigned at Generation Time (Not Extracted)

- **Big Five scores**: Generated by PB&J rationalization to be CONSISTENT with the extracted signals
- **Epistemic boundaries**: Generated based on persona context and role
- **Behavioral guardrails**: Generated based on communication patterns
- **Backstory**: Generated via Anthology-style narrative synthesis (existing pipeline)

This avoids the r<0.26 ceiling entirely by not requiring trait extraction from single transcripts.

### 4.3 Step 3: Simplify and Optimize

**What the pipeline looks like now (7 steps):**

```
1. Upload transcripts (text, any format)                     [NEW]
2. Extract signals (1 LLM call per transcript, parallel)    [NEW]
3. Pool & sample (pure math + 1 validation LLM call)        [NEW]
4. Generate persona (EXISTING GeneratePersonasUseCase)       [EXISTING]
5. Ingest into ID-RAG (EXISTING IdRagStore + interview chunks) [EXTENDED]
6. Chat / simulate (EXISTING ChatAdapter)                    [EXISTING]
7. Tweak & regenerate (slider → modify input → repeat step 4) [NEW]
```

**Extraction simplified to a single LLM call per interview** producing structured JSON with: pain points, goals, values, feature desires, decision patterns, context, communication style, and salient quotes — each with verbatim source quote.

**Pooling simplified to normalized frequency counts** across extraction outputs. No Bayesian inference, no copula modeling.

**Sampling simplified to weighted random draw + one-shot coherence validation**. No MCMC, no rejection sampling loops.

### 4.4 Step 4: Accelerate Cycle Time

| Operation | Serial | Parallel | Target |
|---|---|---|---|
| 20 interviews → extraction | 200s | 15s (parallel) | 15s |
| Pool + sample | <1s | <1s | <1s |
| Coherence validation | 5s | 5s | 5s |
| Generate backstories (40 personas) | 15s | 15s | 15s |
| PB&J + Insights (parallel) | 15s | 10s | 10s |
| ID-RAG ingest | 2s | 2s | 2s |
| Minimal eval | 2s | 2s | 2s |
| **Total (20 → 40)** | **~240s** | **~50s** | **< 60s** |

**Tweak cycle time**: < 5s for a single persona. Only PB&J + system prompt regenerated; backstory cached.

### 4.5 Step 5: Automate

- **Full pipeline triggered by a single server action**: upload → extract → pool → sample → generate → ingest → return cohort
- **Traceability metadata auto-embedded** in every extraction output
- **Psychographic tweaks auto-propagate** to PB&J rationales, anchor text, and system prompt
- **Cohort auto-regenerates on filter changes** with minimal recomputation
- **Minimal coherence check runs automatically** after every generation batch

NOT automated yet: InCharacter/PICon evaluation, activation vector steering, emotional tone analysis.

## 6. Pipeline Flow

### 6.1 Upload

- User uploads interview transcript files (text format, any structure)
- System stores raw text with metadata (filename, upload timestamp, interview count)
- No preprocessing required — raw text goes to extraction

### 6.2 Extraction (Single-Shot, Parallel)

```
For each interview (parallel):
  1. Send raw transcript text to LLM with structured extraction prompt
  2. LLM returns ExtractedInterviewSignals JSON
  3. Store extraction result
  4. Generate InterviewChunks from extraction output (one chunk per signal category)
  5. Return ExtractedInterviewSignals
```

**Prompt structure**: The extraction prompt instructs the LLM to identify specific signal types in the transcript text and extract them verbatim. No trait inference. No psychometric scoring. No summary — just extraction with source quotes.

### 6.3 Pooling

```
Across all ExtractedInterviewSignals outputs:
  1. For each signal type (pain points, goals, etc.):
     a. Normalize similar signals (merge duplicates with fuzzy matching)
     b. Count frequency across interviews
     c. Normalize to weights (0-1)
  2. For context fields:
     a. Aggregate role, industry frequencies
  3. Build pool of all salient quotes
  4. Return PooledDistributionSummary
```

### 6.4 Sampling

```
Given: PooledDistributionSummary, target persona count N, scale multiplier M

For each of N persona slots:
  1. Draw pain points: weighted random from distribution (normally 2-4 per persona)
  2. Draw goals: weighted random (1-3 per persona)
  3. Draw values: weighted random (2-4 per persona)
  4. Draw feature desires: weighted random (1-3 per persona)
  5. Draw decision pattern: weighted random (1 per persona)
  6. Draw context: weighted random role + industry
  7. Assign communication style: weighted random
  8. Collect relevant quotes from pool

After all N samples → 1 coherence validation call:
  "For each persona in this list, flag any internal contradictions: 
   e.g., persona wants low cost AND enterprise-grade features, 
   or persona values simplicity AND wants 50 integrations."

Resample flagged conflicts (<10% expected).
```

### 6.5 Persona Generation (Existing Flow)

Each sampled persona configuration is formatted as a `personaDescription` string and fed into the existing `GeneratePersonasUseCase`:

1. **Brainstorming**: Extract signals → persona profile (existing prompt, now enriched with interview data)
2. **Backstory generation**: Batch call → narrative backstory per persona (existing)
3. **PB&J enhancement**: Psychological rationales generated from extracted signals (existing)
4. **Insight generation**: Behavioral insights (existing)

The Persona entity's fields (Big Five, values, fears, communication style, decision style) are populated by the generation steps, not extracted from transcripts. The extracted interview signals serve as the grounding context that constrains generation.

### 6.6 ID-RAG Ingestion

For each persona:
1. Index persona backstory into IdRagStore (existing flow)
2. Index interview-derived chunks (InterviewChunk objects) alongside backstory chunks
3. Each interview chunk carries: sourceInterviewId, sourceSegmentId, topicTags, signalTypes

### 6.7 User-Facing Modes

**Chat with persona**: Existing ChatAdapter with PersonaPromptCompiler. ID-RAG retrieves both backstory chunks AND interview chunks.

**Cohort simulation**: Iterate over personas, run each through the same scenario prompt, aggregate results. No dedicated cohort engine for MVP — server action loops over personas.

**Tweak psychographics**: User adjusts slider → modified value replaces the PB&J-generated value → persona regenerates with new rationales. Backstory cached.

**Traceability**: When viewing persona responses, user can click "Source" to see the interview excerpt that grounds that claim. Powered by ID-RAG metadata.

## 7. Extraction Schema

### 7.1 Extraction Prompt (One Call Per Interview)

```
You are analyzing a user interview transcript. Extract the following 
observable signals from the text. For each signal, provide:
1. A normalized description
2. The EXACT verbatim quote from the transcript
3. The rough line/segment reference

DO NOT infer personality traits or psychometrics. Only extract what 
is explicitly stated or can be directly supported by quotes.

Extract:
- PAIN POINTS: frustrations, obstacles, difficulties mentioned
- GOALS: what they're trying to achieve, their desired outcomes
- VALUES: what they state is important to them
- FEATURE DESIRES: specific features, capabilities they request
- DECISION PATTERNS: how they describe making decisions
- CONTEXT: their role, industry, company size
- COMMUNICATION STYLE: brief observation of their language patterns

Return as JSON matching the ExtractedInterviewSignals schema.
```

## 8. Pooling and Sampling

### 8.1 Pooling (Pure Function)

```typescript
function poolSignals(allExtractions: ExtractedInterviewSignals[]): PooledDistributionSummary {
  // For each signal type, normalize similar items across interviews
  // using fuzzy text matching, then calculate frequency weights
  // Returns weighted distributions
}
```

No LLM calls. No separate pipeline stage. This runs in <1ms.

### 8.2 Sampling Algorithm

```typescript
function samplePersonas(
  distribution: PooledDistributionSummary,
  personaCount: number
): SampledPersonaSignal[] {
  const personas: SampledPersonaSignal[] = [];
  
  for (let i = 0; i < personaCount; i++) {
    // Weighted random draw for each signal type
    // with_replacement = true (allows multiple personas with similar traits)
    personas.push({
      id: `sampled_${i}`,
      painPoints: weightedDraw(distribution.painPoints, 2, 4),
      goals: weightedDraw(distribution.goals, 1, 3),
      values: weightedDraw(distribution.values, 2, 4),
      featureDesires: weightedDraw(distribution.featureDesires, 1, 3),
      decisionPattern: weightedDraw(distribution.decisionPatterns, 1, 1)[0],
      context: {
        role: weightedDraw(distribution.contextDistribution.roles, 1, 1)[0],
        industry: weightedDraw(distribution.contextDistribution.industries, 1, 1)[0],
      },
      communicationStyle: weightedDraw(distribution.communicationStyles, 1, 1)[0],
    });
  }
  
  return personas;
}
```

### 8.3 Coherence Validation (1 LLM Call)

```
Given this list of N sampled persona signal sets, identify any 
personas where the combination of signals is internally contradictory 
or implausible. Examples of contradictions:
- Persona wants "enterprise-grade everything" but also "free or cheap"
- Persona values "simplicity" but requests "50+ integrations"
- Persona is a "C-suite executive" with "no budget authority"

Return array of persona indices that have contradictions and a 
description of each contradiction. If none, return empty array.
```

## 10. ID-RAG Grounding

### What Gets Indexed

For each persona:
1. **Backstory chunks** (existing behavior — paragraph-level chunks with metadata)
2. **Interview-derived chunks** (new) — InterviewChunk objects created during extraction

### What Gets Retrieved

At chat time, `IdRagService.retrieveContext(persona, query, k=3)` retrieves:
- Backstory chunks semantically relevant to the query
- Interview chunks semantically relevant to the query

### Hybrid Prompt Assembly

The existing `PersonaPromptCompiler.compileInteractionPrompt()` assembles:
1. Compartmentalized persona prompt (identity, psychographics, boundaries, guardrails)
2. `<<RETRIEVED MEMORY>>` section with both backstory and interview excerpts
3. Source attribution: each interview excerpt is tagged with its source interview ID

### Traceability Display

When rendering a persona response that references interview-derived content, the UI shows a citation indicator. User hovers/clicks to see:
> "This response is grounded in interview #3 (Participant A, line 42): [verbatim quote]"

## 11. Psychographic Tweaking

### User Interaction

User sees a persona's psychographic profile (Big Five, values, fears, communication style, etc.) in the persona detail view. Each dimension has a slider.

When the user adjusts a value (e.g., increases price sensitivity from 40 → 70):

### System Response

```
1. Update the modified value in the persona data
2. Re-run PB&J scaffolding with the new value (1 LLM call)
   → Generates new rationale: "This persona is highly price-sensitive because..."
3. Update persona anchor text if affected
4. Return updated persona (no backstory regeneration — cached)
```

**Target latency**: < 5s for a single tweak. Backstory is immutable — only the rationale and prompt change.

### Cohort-Level Tweaks

User sets a filter: "Show me personas with price sensitivity > 60."
- System re-samples from the distribution with the filter applied
- Returns a new cohort from the same pool, filtered
- No regeneration needed — personas already exist in the pool

|---|---|

| Persona pain point | The interview(s) it was drawn from via ID-RAG |
| Persona backstory | Generated (no direct interview mapping) |
